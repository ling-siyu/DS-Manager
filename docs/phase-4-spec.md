# Phase 4 — The gated edit loop (agent contract)

`dsm` ships the deterministic, git-gated primitives of the AI edit loop. **The
brain is the coding agent you are already in** — Claude Code, or any MCP/CLI
agent (Codex, etc.) — which proposes code edits with its own tools and judges
the screenshots with its own vision. **No model API key is ever used**: the
intelligence is the agent session itself (e.g. a Claude Pro/Max subscription
driving Claude Code), not a separate per-token API bill. `dsm` is the
deterministic substrate; the agent is the loop driver.

Every mutation is gated: it happens inside a session pinned to a base commit and
a path scope, and ends in exactly one of **approve** (pathspec commit),
**revert** (restore to base), or **abandon** (keep working tree, end session).

The one-command entry point for Claude Code is the **`/dsm-edit` skill**
(`.claude/commands/dsm-edit.md`), which encodes the recipe below so the user can
just say `/dsm-edit make the buttons pill-shaped`. The same loop is available to
any agent over MCP (`dsm serve`) — see § Driving from an agent.

## The loop

```
dsm edit start [--scope p…] [--json]   # gate: clean tree in scope → session
dsm edit render --label before          # screenshots of affected components
<agent edits files — its own tools, or `dsm scan --fix`>
dsm edit check                          # typecheck changed files; tokens: parse +
                                        #   unresolved-ref validation + rebuild
dsm edit render --label after
dsm edit diff                           # diff.patch + pixelmatch before/after
<agent READS the before/after/diff PNGs → vision verdict>
dsm edit approve -m "…"  |  dsm edit revert  |  dsm edit abandon
```

Every subcommand takes `--json` (machine-readable result, exit code 0/1). The
same operations are exposed as MCP tools by `dsm serve`: `edit_start`,
`edit_status`, `edit_check`, `edit_render`, `edit_diff`, `edit_approve`,
`edit_revert`, `edit_abandon`, `scan_fix`.

## Sessions

- State: `<repoRoot>/.dsm-edit/session.json` (single active session). Artifacts:
  `<repoRoot>/.dsm-edit/<id>/` — `shots/<label>/*.png`, `shots/diff/*.png`,
  `diff.patch`, `diff-report.json`, `result.json`. The `.dsm-edit/` dir is
  self-ignoring (it writes its own `.gitignore`), so it works in any repo.
- Default scope: `design-system/`. **Effective scope auto-extends with
  `design-system/build/`** whenever `tokens.json` is in scope — build artifacts
  are git-tracked and regenerated by token builds, so approve/revert must own them.
- Gates: clean `git status --porcelain` within scope (`--allow-dirty` opts out);
  refuses an unborn HEAD, a second concurrent session (`--force` discards a stale
  record), and any repository other than the one dsm lives in (SecuraMark and
  other `-C` targets stay read-only until the sandbox-branch flow lands).
- `approve` refuses if HEAD drifted from the session base; it commits **only the
  session pathspecs** — unrelated staged changes are untouched.
- `revert` restores tracked files to base and deletes files created during the
  session. Because `build/` is tracked and in the effective scope, no
  post-revert rebuild is needed.

## Rendering

Screenshots use the preview's stage route — `#/stage/<source>/<name>?scenario=<i>`
renders exactly one component scenario with no chrome, kills all
animation/transition, and sets `data-stage-ready="true"` when mounted and fonts
are loaded. `dsm edit render`:

- infers affected components (changed files ∩ registry paths; a `tokens.json`
  change ⇒ all registered components; `--component` overrides),
- boots a fresh preview server (token build included, SecuraMark compile skipped
  for DSM-only targets) on an ephemeral port,
- drives the machine's installed Chrome via puppeteer-core (`CHROME_PATH`
  overrides discovery) with pinned determinism settings (fixed viewport,
  DPR 2, sRGB, no LCD text/hinting, reduced motion, scrollbars hidden),
- shoots every scenario of every affected component into
  `shots/<label>/dsm-<Name>-s<i>.png`, and tears everything down.

`dsm edit diff` writes the unified code diff and a pixel report per shot pair:
`{ shot, changedPct, changedPixels, dimensionsChanged }` — dimension changes
(e.g. a padding edit) are padded and flagged, never fatal. **The verdict is the
agent's job:** read the PNGs, compare against the instruction, then approve or
revert.

## `dsm scan --fix`

Deterministic tokenizer for the unambiguous subset: replaces hardcoded **hex**
colors with `var(--ds-…)` when exactly one token (or exactly one *semantic*
token) carries that value. Hex is normalized (case, 3→6 digits); alpha forms,
CSS selector positions (`#cafe { }`), and ambiguous values are skipped and
reported — `skipped[].candidates` hands the decision to the agent, which knows
the usage context. Requires an active session; only touches files inside the
session scope. rgb()/hsl()/Tailwind-arbitrary violations remain report-only
(their scan regexes capture truncated text; class-level judgment is agent work).

## Failure modes an agent should expect

- `start` fails on dirty scope → commit/stash or `--allow-dirty`.
- `check` exit 1 with diagnostics (`typecheck[]`, `tokens.unresolvedRefs[]`).
- `render` fails fast with the stage's `.render-error` text or the Vite overlay
  message; no Chrome found → set `CHROME_PATH`.
- `approve` fails on HEAD drift → inspect, then `revert`/`abandon` and restart.
- MCP `edit_render` can take 10–30s (Vite + Chrome boot) — call it with patience,
  not retries.

## Driving from an agent (no API key)

The agent **is** the brain — it never calls a model API. Two interfaces:

**Claude Code — the `/dsm-edit` skill.** `.claude/commands/dsm-edit.md` encodes
the full recipe: the user types `/dsm-edit <instruction>`, and Claude Code runs
`start` → `render --label before` → makes the edits itself (tokens-first) with
its own Edit/Write tools → `check` (fixing on failure) → `render --label after`
→ `diff` → **Reads the before/after/diff PNGs with its own vision** → presents
the verdict + screenshots → on the user's say-so, `approve` or `revert`. The
intelligence is the Claude Code session; `dsm` only runs the gates and renders.

**Any MCP agent — `dsm serve`.** The same operations are MCP tools
(`edit_start`, `edit_render`, `edit_check`, `edit_diff`, `edit_approve`, …, plus
`scan_fix`). The driving agent calls the tools and **reads the returned PNG
paths itself** to form the visual verdict. Register it in Claude Code via the
repo's `.mcp.json` (`dsm serve`) — see `docs/mcp-setup.md`. Codex and other
MCP-capable agents drive the identical contract; nothing in the loop is
Claude-Code-specific.

The brain contract, in full: **call the tools in order, make the edits, Read the
shot PNGs, decide approve vs revert.** That is all an agent needs to implement.
